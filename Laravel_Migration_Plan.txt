خطة تحويل مشروع MedSurvey Pro إلى Laravel بدون التأثير على الشكل أو العمل الحالي
=============================================================================

الهدف
-----
تحويل الخادم الخلفي فقط من:

server/  Node.js + Express + Prisma

إلى:

laravel-api/  Laravel API

مع إبقاء واجهة React الحالية كما هي:

src/
public/
index.html
تصميم الواجهات
نظام اللغتين
الشاشات الحالية

الفكرة الأساسية
--------------
لن يتم حذف الخادم الحالي في البداية. سيتم إنشاء مجلد جديد خاص بـ Laravel فقط:

MedSurvey Pro/
├─ src/              واجهة React الحالية
├─ server/           خادم Node الحالي، يبقى كمرجع مؤقت
├─ laravel-api/      خادم Laravel الجديد
├─ public/
├─ package.json
└─ docker-compose.yml

بهذه الطريقة يبقى المشروع الحالي يعمل أثناء النقل، ويمكن مقارنة Laravel مع Node مساراً بمسار.


المرحلة 1: إنشاء Laravel بدون حذف القديم
----------------------------------------
1. إنشاء مجلد جديد باسم laravel-api.
2. تثبيت Laravel داخله.
3. تجهيز إعدادات:
   - MySQL
   - CORS
   - API routes
   - Validation
   - Logging
   - Environment variables
4. إبقاء مجلد server كما هو لاستخدامه كمرجع.


المرحلة 2: نقل قاعدة البيانات
-----------------------------
تحويل Prisma schema إلى Laravel migrations/models مع الحفاظ على أسماء الجداول والأعمدة قدر الإمكان.

الجداول المطلوب نقلها:

- users
- tenants
- surveys
- survey_sections
- survey_questions
- survey_responses
- survey_answers
- tickets
- settings
- audit_logs
- error_logs
- refresh_tokens
- archived_survey_responses
- archived_audit_logs

الهدف هنا هو تقليل التعديلات المطلوبة في الواجهة والحفاظ على توافق البيانات.


المرحلة 3: مطابقة مسارات API الحالية
------------------------------------
بناء نفس المسارات الحالية في Laravel:

- /api/auth
- /api/users
- /api/surveys
- /api/responses
- /api/tickets
- /api/settings
- /api/audit
- /api/monitoring
- /api/backups
- /api/error-logs
- /api/health

يجب أن تكون استجابات JSON قريبة جداً من الخادم الحالي حتى لا تتأثر واجهة React.


المرحلة 4: المصادقة والصلاحيات
-----------------------------
نقل الوظائف التالية:

- login
- refresh token أو طريقة بديلة مناسبة
- logout
- /auth/me
- حماية المسارات
- الأدوار:
  - super_admin
  - admin
  - unit_manager
  - head_of_department
  - staff
- صلاحيات إدارة المستخدمين والاستبيانات والتقارير.

هذه المرحلة حساسة لأنها تؤثر على الدخول ولوحة التحكم.


المرحلة 5: نقل الوظائف الأساسية بالتدريج
----------------------------------------
يتم النقل بالترتيب التالي:

1. settings
2. surveys
3. responses
4. stats/reports
5. tickets
6. users
7. audit logs
8. error logs
9. backups
10. monitoring

بعد كل جزء يتم اختباره قبل الانتقال للجزء التالي.


المرحلة 6: تعديل React بأقل قدر ممكن
------------------------------------
الواجهة يجب أن تبقى كما هي.

قد نحتاج فقط تعديل ملف الاتصال بالـ API:

src/api/core.ts

خصوصاً إذا تغيرت طريقة المصادقة أو CSRF.

الصفحات التي يجب ألا يتغير شكلها:

- Dashboard
- SurveyBuilder
- Responses
- Tickets
- Users
- Settings
- Reports
- Backups
- Audit Logs
- Survey pages
- Login page


المرحلة 7: الاختبار شاشة شاشة
-----------------------------
اختبار الوظائف التالية:

- تسجيل الدخول
- تسجيل الخروج
- استرجاع بيانات المستخدم الحالي
- إنشاء مستخدم
- تعديل مستخدم
- تغيير كلمة المرور
- إنشاء استبيان
- تعديل استبيان
- حذف استبيان
- تعبئة استبيان كمريض
- ظهور الرد في لوحة التحكم
- توليد التذاكر
- تعديل حالة التذكرة
- الفلاتر في الردود والتقارير
- الإحصائيات
- الصلاحيات حسب الدور
- الإعدادات
- سجلات التدقيق
- سجلات الأخطاء
- النسخ الاحتياطي إذا كانت الاستضافة تدعمه


المرحلة 8: تجهيز النشر على استضافة مشتركة
-----------------------------------------
تجهيز Laravel ليستطيع العمل على shared hosting أو cPanel:

- إعداد MySQL من لوحة الاستضافة.
- رفع ملفات Laravel.
- توجيه document root إلى مجلد public في Laravel.
- تشغيل composer install إذا كانت الاستضافة تدعم Composer.
- إعداد ملف .env.
- تشغيل php artisan key:generate.
- تشغيل migrations.
- إعداد storage permissions.
- إنشاء storage link إذا لزم.
- إعداد cron job للمهام المجدولة إن كانت مطلوبة.

واجهة React يمكن نشرها بطريقتين:

1. بناء React ورفع الملفات الثابتة إلى نفس الاستضافة.
2. نشر React على Cloudflare Pages أو Netlify وربطها بـ Laravel API.


المرحلة 9: إيقاف Node بعد التأكد
--------------------------------
بعد التأكد أن Laravel يغطي كل الوظائف:

1. تشغيل الاختبارات النهائية.
2. مقارنة النتائج مع الخادم القديم.
3. الاحتفاظ بنسخة أرشيفية من server.
4. تحويل الواجهة نهائياً إلى Laravel API.
5. إيقاف خادم Node.


القرار التقني المقترح
--------------------
التركيبة الموصى بها:

React frontend
Laravel API backend
MySQL database
بدون Redis في البداية

سبب تجنب Redis في البداية:
الاستضافات المشتركة غالباً لا تدعمه. يمكن استخدام database/file cache في Laravel بدلاً منه.


تقدير الوقت
-----------
تقدير واقعي للتحويل الصحيح:

- نسخة Laravel API أساسية: 1 إلى 2 أسبوع
- نقل الوظائف المهمة: 3 إلى 5 أسابيع
- اختبار وتجهيز الاستضافة: 1 إلى 2 أسبوع

الإجمالي المتوقع: 4 إلى 7 أسابيع.


ملاحظات مهمة
------------
1. لا يتم حذف server في البداية.
2. لا يتم تغيير شكل الواجهة.
3. يتم النقل بالتدريج.
4. كل API يتم اختباره قبل الانتقال لغيره.
5. الهدف الأساسي من التحويل هو تقليل تكلفة الاستضافة، وليس تغيير تجربة المستخدم.
6. إذا وُجدت استضافة رخيصة تدعم Node.js جيداً، قد لا يكون التحويل ضرورياً.
